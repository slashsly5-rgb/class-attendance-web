# Phase 5 Location-Based Verification: Implementation Plan

## Objective
Implement geolocation verification to ensure students are physically present in the classroom when submitting their attendance, utilizing browser capabilities to prompt the student and test against the class config.

## User Review Required
None immediately blocking. The Haversine algorithm is standard mathematics.

## Proposed Changes

### Utilities
#### [NEW] `src/lib/utils/geolocation.ts`
- `calculateDistance(lat1, lon1, lat2, lon2)`: Mathematical implementation of the Haversine formula returning real-world distance in meters.
- `useGeolocation()`: A custom React hook managing the browser's `navigator.geolocation` API, returning `{ coordinates, error, requestPermission, isLoading }` state properties.

### Components
#### [NEW] `src/components/features/attendance/LocationVerification.tsx`
- A client component heavily relying on `useGeolocation`.
- On mount, it should idle or request permission dependent on initial configuration.
- Displays appropriate error handling UI components for:
  - User explicitly denied permission.
  - GPS hardware unavailable or disabled.
  - Location acquired, but mathematical distance exceeds the class `radius`.
- Provides an input field or button sequence explicitly for an "Emergency Bypass / Override code" if a single user's device fails heavily. The bypass logic itself could be a simple server-validated PIN or similar.

## Verification Plan
### Automated Tests
- Run `npx tsc --noEmit` to verify type safety.

### Manual Verification
- Will test the location calculation against hard-coded mocked GPS coordinates locally to simulate being "inside" vs "outside" the radius of an established classroom.
