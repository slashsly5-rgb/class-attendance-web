# Phase 7 Retroactive Attendance: Implementation Plan

## Objective
Enable lecturers to securely grant targeted students the ability to submit attendance for a session that has already been closed. Integrate the UI and Server Actions to track and consume these retroactive access passes.

## User Review Required
None critical. We will reuse `shadcn/ui` components for the selection list. 

## Proposed Changes

### 1. Server Actions & Validation
#### [NEW] `src/lib/validations/retroactive.ts`
- `grantAccessSchema`: Validates an array of `student_id` strings and a single `session_id`.

#### [NEW] `src/lib/actions/retroactive.ts`
- `grantRetroactiveAccess`: Inserts rows into the `retroactive_access` table for the selected students, defaulting `used` to FALSE.
- `getRetroactiveAccessBySession`: Returns a list of which students have been granted access to a specific session so the lecturer knows the state.

#### [MODIFY] `src/lib/actions/attendance.ts`
- **Modify `getActiveSessionForClass`**: Rename its logical use or create a new wrapper `getStudentSessions(studentId)` that returns both globally active sessions AND sessions where this student has an unused `retroactive_access` pass.
- **Modify `submitAttendance`**: 
  - Enhance the activity check: Allow submission if session is globally active OR if there is an unused row in `retroactive_access` for this `(session_id, student_id)`.
  - If fulfilling a retroactive pass, update the `retroactive_access` row to `used = true` and insert the `attendance_records` row with `is_retroactive = true`.

### 2. Lecturer UI
#### [NEW] `src/app/lecturer/classes/[id]/sessions/[sessionId]/page.tsx`
- We currently don't have a "Session Detail" page. We will build a focused page specifically for managing a single attendance session.
- Displays the session stats (Total Present, Late, Absent).
- Features a **Retroactive Access Management** card:
  - List of students enrolled in the class who DO NOT have an attendance record for this session.
  - Checkboxes to select students and a "Grant Access" button.

#### [MODIFY] `src/app/lecturer/classes/[id]/page.tsx`
- Update the `SessionList` component. The active/close buttons remain, but each session row now becomes a clickable link routing to `/lecturer/classes/[id]/sessions/[sessionId]`.

### 3. Student UI
#### [MODIFY] `src/app/student/dashboard/page.tsx`
- Modify the display logic: If the class session array returned from the modified `getStudentSessions` contains a retroactive pass, badge it differently (e.g., "RETROACTIVE ACCESS GRANTED" in amber instead of the green "ACTIVE NOW" pulse).

## Verification Plan
1. Log in as a lecturer. Close an active session. navigate into it.
2. Grant a specific mock student retroactive access.
3. Switch profiles to the mocked student. View the dashboard to verify the amber badge appears.
4. Submit attendance using the override gate. 
5. Verify the DB marks `used = true` and the pass disappears from the dashboard.
